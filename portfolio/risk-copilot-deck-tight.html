WHY THIS MCP SERVER

What the server actually is

LLM

Claude, GPT, Llama — speaks MCP, holds no credentials

qualys-cli-mcp

Validates argv, redacts secrets, enforces policy, writes the audit log

qualys-cli

Auth, retries, the actual Qualys API calls

Qualys Cloud

VM · PC · PM · WAS · TotalCloud · CS · CSAM · Cloud Agent

The server never touches Qualys directly. It wraps an existing CLI and puts a governance layer in front of it — which is precisely why it can be productised rather than demoed.
WHY THIS MCP SERVER

Three unlocks

01

One thread, twelve modules

Natural language replaces hand-built QQL across VM, PC, PM, WAS, TotalCloud, Container Security, CSAM and Cloud Agent.

02

No API client to build

Sixteen typed tools plus a generic passthrough. The integration cost of a cross-module workflow collapses to a config entry.

03

Guardrails in the box

Argv validation, keyring credentials, module allowlist, read-only mode, append-only JSONL audit — shipped, not bolted on.

DEEP DIVE

The problem, in the customer's words

01

“Nothing tells us what to fix first.”

Hosts, cloud resources, containers and web apps each rank separately. There is no single list.

02

“We join it by hand.”

An analyst opens six modules to answer one question about one asset. Every week.

03

“The Monday report is a spreadsheet.”

Defensible to leadership, provable to nobody. No record of what was pulled, or when.

04

“Engineering wants answers, not access.”

Managers need to know how risky their estate is — without a Qualys seat or write permissions.

The bespoke ask — qualys_cli()

One asset. Five modules. One story.

Expand prod-checkout-web-01 (TruRisk 96) and the cross-module chain unfolds in place.
  • VM — Log4Shell, CVSS 10.0, KB confirms active exploitation.
  • WAS — reflected SQL injection on /checkout/apply-promo.
  • PC — fails CIS 5.2.9, SSH root login is enabled.
  • CA — agent healthy, so this score can be trusted.
  • PM — a remediation job exists and has not run. Expedite it.
